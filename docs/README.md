#### 業務の中で工夫したこと、成果・実績

#### 1. Component のリファクタリング

#### 目的

- Presentational Component と Redux が密結合して、Storybook、React Testing Library で統合テストを書く際に store を再定義する工数がかかる。開発スピード・開発体験を良くするために Redux と疎結合にする
- Component を条件分岐で出し分けていて、論理的凝集になっている Component があるので、可読性・保守性・再利用性を改善する

#### 結果

- 一部の Component を Redux と疎結合にすることで Storybook、React Testing Library での開発スピード・開発体験が良くなった。また、Redux への依存性がなくなったことで、他のライブラリへの移行をしやすくなった
- 一部の Component を単一責任に分離することで、可読性・保守性・再利用性が改善した

#### 使用した技術

- React
- Redux

#### どうやったか

- Presentational Component と Redux の connect をしている箇所を排除し、必要なデータは Container Component に寄せた。Container Component から対象の Presentational Component にバケツリレーで props を渡す
- Component の責務が複数になっている箇所を特定して、単一責任になるように Component を分離する

#### 工夫した点

- リファクタリングはバグを生み出す可能性もあるので、リファクタリング対象にテストがなければ E2E で正常動作のテストを追加した
- リファクタリングでもレビュアーがレビューしやすいように、そしてバグが生じた際の原因特定のしやすさの観点からプルリクエストの目的は 1 つにして粒度を小さくするようにした
- Component を共通化する場合は、Component が使われる or 変更される目的は 1 つになっているかを意識した

#### 残った課題と解決策

- Container Component に必要なデータを寄せると Container Component が肥大化して見通しが悪くなる。ロジックを Hooks 化 or Clean Architecture の設計を採用して見通しをよくする必要がある
- Component を単一責任に分離しても、一覧性が良くなければ再利用しづらい。Storybook or Atomic Design を使用して一覧性を改善する必要がある

#### 2. 担当者選択欄の UI 変更

#### 目的

- 担当者を検索できる機能があるが検索できるかがわかりにくく、機能に気づかないユーザーがいた。UI を変更して機能を気づきやすいようにしたい

#### 結果

- UI を変更して検索機能があることをわかりやすくした
- 当初は他のタスクで検索欄を追加する予定であったが、機能を追加する必要性がなくなった

#### 使用した技術

- React
- Redux
- React Select

#### どうやったか

- 1 担当者選択欄の input をクリックした後のカーソル位置がアイコンと被っている
- 2 担当者選択欄に選択されているユーザーがいた場合、input をクリックした後もユーザー名が input に残る
- 上記 2 点の問題があったので、1 に関してはカーソル位置の修正。2 に関してはユーザー名の色を黒から薄いグレーにすることで、文字を入力できることを示唆するようにした

#### 工夫した点

- 他のサービスの選択欄がどのような UI になっているか調査した
- なぜわかりにくいかを深掘りした
- 選択欄は React Select を使用していたので、できる限りライブラリで完結できるようにした

#### 3. セクション機能の開発（実装中）

#### 目的

- チェックリストは親子関係の 2 階層まで作成できるが、3 階層まで作成したい要望が多かった。セクションでグルーピングをできるようにする

#### 使用した技術

- React
- Redux
- React Sortable Tree

#### どうやったか

- 使用していた React Sortable Tree を活用することで、セクションによるグルーピング機能を実現した
- セクションとチェックリストのテーブルは分けて実装した
- 状態管理は Redux を使用した

#### 工夫した点

- 使用していたライブラリを活用した
- React Sortable Tree は 3 階層まで作成できるので 、1 つのテーブルで管理することも可能だったが、下記 2 点からテーブルを分離することにした
  - セクションとチェックリストでカラムが異なるので null を許容する必要がある
  - テーブルの構造を理解しにくくなる
- 大きめで複雑な実装なので設計段階から相談を重ねた。また、プルリクの粒度を小さくして頻繁にレビューを通すことで、すぐにフィードバックをもらえるようにした

#### 残った課題と解決策

- セクションとチェックリストの階層構造を保つためには同じ Reducer で管理する必要性があったので、Reducer が肥大してしまい見通しが悪い。使用しているロジックも utilities フォルダーにまとめて置いているので見通しが悪い
- Re-ducks パターンを採用して保守性・可読性を改善する
